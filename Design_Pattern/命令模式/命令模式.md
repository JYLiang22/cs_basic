# 命令模式结构图

![Alt text](%E7%BB%93%E6%9E%84%E5%9B%BE.png)<br>

1. Command(抽象命令类)：声明了用于执行命令的接口Execute()
2. ConcreteCommand(具体命令类)：实现执行命令类的接口Execute()，它对应具体的接收者对象，将接收者绑定其中
3. Invoker(调用者)：请求的发送者，通过命令对象来执行请求，一个调用者不需要在设计时确定其接收者，所以调用者通过聚合与命令类产生关联。具体实现中，可以将一个具体命令对象注入到调用者中，再通过调用具体命令对象的Execute()方法，实现接收者的操作
4. Receiver(接收者)：实现接收请求的具体操作

# 优缺点

优点：

1. 降低系统耦合度，将命令的请求者和接收者分离解耦，请求者和发送者不存在直接关联，各自独立互不影响
2. 便于拓展，新的命令很容易加入到系统中，符合开闭原则
3. 较容易实现命令队列
4. 为请求的撤销和恢复操作提供了一种设计实现方案

缺点：

1. 命令模式可能导致系统中有过多的具体命令类，增加了系统中对象的数量

适用环境：

1. 系统需要将请求发送者和接收者解耦，使得发送者和接收者互不影响
2. 系统需要在不同时间指定请求、将请求排队和执行请求
3. 系统需要支持命令的撤销和恢复操作
4. 系统需要将一组操作组合在一起形成宏命令
